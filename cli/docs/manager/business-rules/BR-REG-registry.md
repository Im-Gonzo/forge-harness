# Business Rules â€” Registry & inventory (BR-REG)

> The normative rules for the global artifact catalog. The registry is the manager's spine: identity,
> status, and the reverse-index every other dimension reads. Decided by ADR-0005, ADR-0006, ADR-0008,
> ADR-0014; detailed by SPEC-01; dependency-edge rules live in BR-DEP; per-artifact version semantics in
> BR-VER. **Phase: v0.2** unless a rule notes otherwise.

### BR-REG-001 â€” Registry location and ownership

**Rule:** The global registry MUST live at `forge/.forge/registry.json` (git-tracked) with a sibling
append-only `forge/.forge/registry.log.jsonl`. The manager MUST NOT introduce a separate lock file. The
on-disk *location and storage policy* of `forge/.forge/` is owned by Bundle F (see ADR-0003, BR-INT); this
rule only fixes the registry's path and the no-lock-file constraint.
**Rationale:** Identity must be versioned with the code it describes (git-tracked truth, C6). JSONL gives
an audit trail without a lock; flat JSON + JSONL is the storage engine (ADR-0002).
**Acceptance:** `forge registry build --write` creates exactly `registry.json` and (on mutation)
`registry.log.jsonl` under `forge/.forge/`, and no `*.lock` file â€” `EVAL-REG-001`.
**Priority:** MUST
**Refs:** ADR-0002, ADR-0003, ADR-0008, SPEC-01

### BR-REG-002 â€” Generated, never hand-edited

**Rule:** The registry MUST be generated by scanning the library and MUST NOT be hand-edited. `forge
registry build` MUST be the only writer. Staleness is **split by drift kind**: a committed registry that
differs **structurally** from a fresh rebuild â€” a uid added/removed, or a shared artifact whose
`{kind,id,path,status,modules}` changed â€” is **stale (ERROR)**; a **content-only** change (same structural
identity, only `contentHash` differs) is the advisory bump-gate WARN (BR-VER-006), never a stale ERROR.
**Rationale:** The live-symlink seam (ADR-0008) means content changes with no event; only a deterministic
scan reflects reality. Structural drift means the catalog itself is wrong (block); a lone content edit is
just an un-bumped revision (advise, don't deadlock the solo dev â€” ADR-0007).
**Acceptance:** `lint/validate-registry.mjs` rebuilds in memory and, on **structural** drift, emits ERROR
`"registry stale, run forge registry build --write"` and exits 1 â€” `EVAL-REG-002`.
**Priority:** MUST
**Refs:** ADR-0008, ADR-0014, SPEC-01

### BR-REG-003 â€” Library scan surface and kindâ†’path resolution

**Rule:** The scan MUST cover `agents/ skills/ commands/ rules/ hooks/ bundles/`, validators in `lint/`,
meta-tests in `tests/meta/`, and engine scripts under `bootstrap/`. It MUST resolve each artifact's
`kind` and `path` using the **same** mapping as `validate-manifests.mjs` (`agents/<id>.md`,
`skills/<id>/SKILL.md`, `commands/<id>.md`, `rules/**/<id>.md`, `bundles/<id>.md`,
`lint/<id>.mjs`, engine `bootstrap/<id>.mjs`), via shared `forge/manager/lib/resolve-kind.mjs`. **Hooks
MUST resolve to ids declared in `hooks/hooks.json`, not to files.**
**Rationale:** Reusing the existing resolver guarantees the registry and the composition validator agree
about what exists; divergent resolution would make the registry lie about the very files forge validates.
**Acceptance:** A fixture library produces one record per real artifact across all listed kinds, with hooks
recorded by id; the kind/path of each record matches what `resolve-kind.mjs` yields â€” `EVAL-REG-003`.
**Priority:** MUST
**Refs:** ADR-0008, SPEC-01, SPEC-03

### BR-REG-004 â€” Per-artifact record shape

**Rule:** Each record MUST carry: `uid` (`"<kind>:<id>"`), `kind`, `id`, `path`, `contentHash` (sha256,
per ADR-0005), `revision` (monotonic int), `version` (semver), `status`
(`active|deprecated|experimental|planned`), `criticality` (`safety|compliance|normal`), `owner`,
`description`, `tags[]`, `modules[]` (reverse-index of `modules.json`), `dependsOn[]` (uids; see BR-DEP),
`eval{}` (linkage slot), `createdAt`, `updatedAt`. The registry MUST be authoritative for `version`/
`revision`/`contentHash` (BR-VER); it merely **stores** `criticality`, whose semantics are owned by
Bundle D (ADR-0013).
**Rationale:** A fixed shape is what every other dimension reads; `modules[]` and `dependsOn[]` make the
catalog a graph, not a list. Storing (not defining) `criticality` keeps the safety-lock concept in Bundle D.
**Acceptance:** Every emitted record validates against the SPEC-01 schema with all listed fields present and
correctly typed; `modules[]` equals the reverse-index of `modules.json` for that artifact â€” `EVAL-REG-004`.
**Priority:** MUST
**Refs:** ADR-0005, ADR-0006, ADR-0013, SPEC-01

### BR-REG-005 â€” Planned vs orphan classification

**Rule:** An artifact named in a manifest but with **no file on disk** MUST be recorded with
`status: "planned"` (the `validate-manifests` "(planned)" set). An artifact **on disk but in no module**
MUST be flagged as an **orphan** (see BR-DEP-006 for the inbound-edge refinement). The two are distinct:
planned = declared-not-built; orphan = built-not-declared.
**Rationale:** Forge's Phase-1 manifests are full of planned components; conflating "planned" with
"missing/error" would make the registry scream on a healthy design-stage tree.
**Acceptance:** Given a manifest component with no file, its record has `status: "planned"` and is **not**
an error; given a file in no module, it is reported as an orphan, not as planned â€” `EVAL-REG-005`.
**Priority:** MUST
**Refs:** ADR-0008, SPEC-01, BR-DEP

### BR-REG-006 â€” Idempotent, deterministic build

**Rule:** `forge registry build` MUST be deterministic: two consecutive builds of an unchanged tree MUST
produce byte-identical `registry.json` (stable `uid` sort, stable key order) and MUST append **zero** new
`registry.log.jsonl` lines on the second run.
**Rationale:** Determinism is what makes the staleness check (BR-REG-002) and the version fold (BR-VER) a
clean equality rather than a diff that flaps.
**Acceptance:** `build --write` twice on the same tree yields identical files and no second log line â€”
`EVAL-REG-006`.
**Priority:** MUST
**Refs:** ADR-0008, SPEC-01

### BR-REG-007 â€” Mutation log entries

**Rule:** Every accepted change to a record (new artifact, hash/revision/version change, status change)
MUST append one `registry.log.jsonl` line `{ts, uid, from{hash,rev,ver}, to{...}, reason, evalStatus}`.
The log MUST be append-only; existing lines MUST NOT be rewritten.
**Rationale:** The per-artifact changelog (BR-VER) *is* the log filtered by `uid`; an append-only trail is
the audit primitive that replaces a lock file.
**Acceptance:** Bumping an artifact appends exactly one well-shaped line whose `from`/`to` match the record
before/after, and prior lines are unchanged â€” `EVAL-REG-007`.
**Priority:** MUST
**Refs:** ADR-0002, ADR-0006, SPEC-01, BR-VER

### BR-REG-008 â€” VERSION triple-drift assertion

**Rule:** `validate-registry` MUST assert `forge/VERSION` equals `package.json.version` equals the
`plugin.json` version **after stripping a `-design` suffix** from each (per `bin/forge.mjs#forgeVersion`).
Drift = the stripped **core** versions differ; only then MUST the check flag it, naming all three raw values.
Because the strip makes `0.1.0-design` and `0.1.0` the same release, the current tree
(`0.1.0-design`/`0.1.0`/`0.1.0`) is **aligned** and MUST NOT be flagged.
**Rationale:** A real core-version mismatch is silent and worth surfacing (ADR-0006); a mere pre-release
suffix is not a mismatch. Whether the drift WARNs or ERRORs is set by BR-VER's roll-up rule.
**Acceptance:** On the drifted fixture (`VERSION` `0.2.0` vs `0.1.0`/`0.1.0`) the check reports the drift
with all three raw values; on the real tree (`-design` stripped) it reports **no** drift â€”
`EVAL-REG-008` (and `EVAL-VER-001`).
**Priority:** MUST
**Refs:** ADR-0006, SPEC-01, SPEC-02, BR-VER

### BR-REG-009 â€” Query verbs

**Rule:** The registry MUST expose `forge registry build [--write] | ls [--kind <kind>] | show <uid> |
changed [--since <ref>] | diff <refA> <refB>`. `ls`/`show`/`changed`/`diff` MUST be read-only.
`deps`/`rdeps`/`orphans`/`dangling` are owned by SPEC-03 (BR-DEP), not this rule.
**Rationale:** These are the inventory verbs `forge status` and humans compose; read-only by default honors
the dry-run contract (C4).
**Acceptance:** `ls --kind agents` lists only agent records; `show <uid>` prints one record;
`changed --since <ref>` lists artifacts whose `revision` advanced since `ref`; none of these write â€”
`EVAL-REG-009`.
**Priority:** MUST
**Refs:** ADR-0015, SPEC-01

### BR-REG-010 â€” Registry emits standard findings; build is fail-open

**Rule:** Registry checks MUST emit findings in the standard `{level, path, line, message, source}` shape
(C2) with `source: "validate-registry"`. A scan error on a single malformed artifact MUST be reported as a
finding and MUST NOT abort the whole build (fail-open, invariant 4).
**Rationale:** Uniform findings feed the `--json` envelope (ADR-0004); fail-open keeps one bad file from
blinding the catalog.
**Acceptance:** Given one unreadable/malformed artifact among valid ones, `build` records the rest and
emits a single finding for the bad one in the standard shape â€” `EVAL-REG-010`.
**Priority:** SHOULD
**Refs:** ADR-0004, ADR-0014, C2, SPEC-01, BR-INT
